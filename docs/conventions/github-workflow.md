# GitHub 작업 흐름 규칙

## 목적

이 문서는 대규모 문서 이미지 전처리 플랫폼의 이슈, 브랜치, 커밋, PR 작업 규칙을 정의한다.

## 기본 원칙

1. `main` 브랜치에 직접 push하지 않는다.
2. 이슈 없이 브랜치를 만들거나 작업하지 않는다.
3. 하나의 PR은 하나의 작업 단위만 포함한다.
4. PR 생성 후 Codex가 임의로 merge하지 않는다.
5. 명세는 PR 본문에만 두지 않고 `docs/` 하위 Markdown 문서로 저장한다.
6. 가능한 범위에서 테스트 또는 빌드를 실행하고 PR 본문에 결과를 남긴다.

## 표준 작업 순서

1. 작업 범위를 확인한다.
2. 구현 단위로 GitHub Issue를 생성한다.
3. 이슈 번호 기반 브랜치를 생성한다.
4. 해당 브랜치에서 작업한다.
5. 관련 명세 문서를 `docs/` 하위에 저장하거나 갱신한다.
6. 테스트 또는 빌드로 검증한다.
7. 커밋한다.
8. 원격 브랜치로 push한다.
9. PR을 생성한다.
10. 리뷰어가 확인해야 할 부분을 PR 본문에 적는다.
11. 사용자 검토를 기다린다.

## 브랜치 컨벤션

브랜치는 아래 형식을 사용한다.

```text
타입/닉네임/이슈번호
```

예시:

```text
docs/som/1
chore/som/2
feat/som/3
fix/som/4
test/som/5
ci/som/6
```

사용 가능한 타입:

| 타입 | 의미 |
|---|---|
| `feat` | 새 기능 |
| `fix` | 코드 수정 |
| `bug` | 버그 수정 |
| `refactor` | 리팩토링 |
| `style` | 포맷팅, 코드 스타일 |
| `docs` | 문서 |
| `test` | 테스트 |
| `chore` | 설정, 빌드, 기타 작업 |
| `hotfix` | 긴급 수정 |
| `perf` | 성능 개선 |
| `ci` | CI/CD |

## 이슈 제목 컨벤션

```text
:이모지: [타입] 제목
```

예시:

```text
📝 [Docs] GitHub workflow, code convention, API response convention 문서 추가
🧹 [Chore] 모노레포 기본 디렉터리 생성
✨ [Feat] Job 도메인 skeleton 추가
```

## PR 제목 컨벤션

PR 제목도 이슈 제목과 같은 형식을 사용한다.

```text
:이모지: [타입] 제목
```

## 커밋 메시지 컨벤션

커밋 메시지는 아래 형식을 사용한다.

```text
[타입] 내용
```

예시:

```text
[Docs] 작업 규칙 문서 추가
[Chore] 모노레포 기본 디렉터리 생성
[Feat] Job 도메인 엔티티 skeleton 추가
```

## 작업 단위 기준

좋은 작업 단위:

1. GitHub workflow 문서 추가
2. 공통 응답과 예외 처리 skeleton 추가
3. auth 도메인 기본 구조 생성
4. project 도메인 skeleton 추가
5. worker listener skeleton 추가
6. NGINX routing skeleton 추가

나쁜 작업 단위:

1. 인증, 업로드, Job, Worker, NGINX를 한 PR에 모두 구현
2. 구조 개편과 기능 구현과 버그 수정을 한 PR에 섞기
3. 문서 작업과 대규모 리팩토링을 한 PR에 섞기

## Definition of Done

하나의 PR은 아래 조건을 만족해야 완료로 본다.

1. 이슈가 존재한다.
2. 브랜치명이 컨벤션을 따른다.
3. 작업 범위가 하나의 구현 단위다.
4. 관련 문서가 `docs/` 하위에 저장 또는 갱신되었다.
5. 가능한 테스트 또는 빌드 검증 결과가 기록되었다.
6. PR 본문에 리뷰어가 확인해야 할 부분이 적혀 있다.
7. Codex가 임의로 merge하지 않았다.
