# CODEX HARNESS: 이슈·브랜치·PR·문서화 작업 규칙

> 이 파일은 Codex가 프로젝트 작업을 수행할 때 반드시 따라야 하는 하네스 문서다.  
> 프로젝트는 **대규모 문서 이미지 전처리 플랫폼**이며, Spring 도메인형 구조와 작업 단위 PR 방식을 따른다.

---

## 0. 프로젝트 전제

이 프로젝트는 단순 이미지 리사이징 서비스가 아니다.

목표는 다음과 같다.

- 사용자가 대량의 스캔 문서 이미지를 업로드한다.
- Spring Boot REST API가 인증, 업로드, 작업 등록, 상태 조회를 담당한다.
- RabbitMQ를 통해 이미지 전처리 작업을 Worker에 비동기로 전달한다.
- Worker는 image-test 레포의 OpenCV 기반 문서 이미지 전처리 메커니즘을 사용한다.
- 처리 결과, preview, debug artifact, processing report를 Object Storage에 저장한다.
- PostgreSQL에는 사용자, 프로젝트, 이미지, 작업, 처리 상태, 리포트 메타데이터를 저장한다.
- NGINX는 프론트와 Spring API의 단일 진입점 역할을 한다.
- Google, Kakao 소셜 로그인을 지원한다.

---

## 1. 절대 금지 사항

Codex는 아래 항목을 절대 하지 않는다.

1. `main` 브랜치에 직접 push하지 않는다.
2. 이슈 없이 브랜치를 만들거나 작업하지 않는다.
3. 여러 기능을 하나의 거대한 PR에 몰아넣지 않는다.
4. 리뷰 없이 PR을 merge하지 않는다.
5. API 서버에 OpenCV 전처리 작업을 직접 넣지 않는다.
6. Worker에 로그인, 사용자 인증, 화면용 API 로직을 넣지 않는다.
7. Spring 구조를 최상위 `controller`, `service`, `repository`, `dto` 계층형으로 만들지 않는다.
8. Bootstrap, jQuery, AdminLTE 같은 임의 UI 템플릿을 추가하지 않는다.
9. 명세를 PR 본문에만 적고 끝내지 않는다.
10. 실행 검증 없이 PR을 올리지 않는다.

## 2. 기본 작업 흐름

모든 작업은 아래 순서로 진행한다.

```text
1. 작업 범위 확인
2. 구현 단위로 이슈 생성
3. 이슈 번호 기반 브랜치 생성
4. 해당 브랜치에서 작업
5. 관련 명세 문서를 docs/ 하위에 저장 또는 갱신
6. 테스트 및 실행 검증
7. 커밋
8. 원격 브랜치 push
9. PR 생성
10. 리뷰어 지정
11. 사용자 검토 대기
12. 리뷰 반영
13. 리뷰 승인 후 merge
```

Codex는 PR을 생성한 뒤 사용자에게 검토받아야 한다.  
Codex가 임의로 merge하지 않는다.

## 3. 구현 단위 기준

하나의 이슈와 하나의 PR은 아래 기준 중 하나를 만족해야 한다.

### 좋은 작업 단위

- `auth` 도메인 기본 구조 생성
- Google OAuth2 로그인 skeleton 추가
- Kakao OAuth2 로그인 skeleton 추가
- 프로젝트 생성/조회 API 추가
- 업로드 세션 API 추가
- presigned URL 발급 기능 추가
- Job 생성 API 추가
- RabbitMQ 메시지 발행 기능 추가
- Worker 메시지 listener skeleton 추가
- 전처리 pipeline step skeleton 추가
- MinIO storage adapter 추가
- NGINX reverse proxy 설정 추가
- Docker Compose local 환경 추가
- Swagger 공통 응답 형식 적용
- 공통 예외 응답 구조 추가
- 특정 도메인 테스트 추가

### 나쁜 작업 단위

- 인증, 업로드, Job, Worker, NGINX를 한 PR에 모두 구현
- 구조 개편과 기능 구현과 버그 수정을 한 PR에 섞기
- 문서 작업과 대규모 리팩토링과 API 구현을 한 PR에 섞기
- 단순 오타 수정과 기능 추가를 같은 PR에 넣기

## 4. 브랜치 컨벤션

브랜치는 반드시 이슈에서 파생한다.

```text
타입/닉네임/이슈번호
```

예시:

```text
feat/som/1
fix/som/2
refactor/som/3
docs/som/4
test/som/5
chore/som/6
ci/som/7
```

사용 가능한 타입:

| 타입 | 의미 |
|---|---|
| `feat` | 새 기능 |
| `fix` | 코드 수정 |
| `bug` | 버그 수정 |
| `refactor` | 리팩토링 |
| `style` | 포맷팅, 코드 스타일 |
| `docs` | 문서 |
| `test` | 테스트 |
| `chore` | 설정, 빌드, 기타 작업 |
| `hotfix` | 긴급 수정 |
| `perf` | 성능 개선 |
| `ci` | CI/CD |

## 5. 커밋 컨벤션

커밋 메시지는 아래 형식을 사용한다.

```text
[타입] 내용
```

예시:

```text
[Feat] Job 도메인 엔티티와 상태 enum 추가
[Fix] OAuth2 리다이렉트 URI 설정 수정
[Bug] 업로드 세션 완료 처리 오류 수정
[Refactor] Project 권한 검증 로직 분리
[Style] 코드 포맷팅 정리
[Docs] API 명세 문서 추가
[Test] Job 생성 서비스 테스트 추가
[Chore] Docker Compose 로컬 설정 추가
[Hotfix] 운영 OAuth callback 오류 긴급 수정
[Perf] 이미지 목록 조회 쿼리 개선
[Ci] backend-api CI workflow 추가
```

커밋 타입:

| 타입 | 의미 |
|---|---|
| `[Feat]` | 새로운 기능 추가 |
| `[Fix]` | 코드 수정 |
| `[Bug]` | 버그 수정 |
| `[Refactor]` | 기능 변화 없는 리팩토링 |
| `[Style]` | 포맷팅, 세미콜론, 들여쓰기 등 |
| `[Docs]` | README, 명세, 주석 등 문서 수정 |
| `[Test]` | 테스트 코드 추가/수정 |
| `[Chore]` | 빌드, 패키지, 설정 등 |
| `[Hotfix]` | 급한 에러 처리 |
| `[Perf]` | 성능 개선 |
| `[Ci]` | CI/CD |

## 12. 리뷰 규칙

PR을 올릴 때 반드시 리뷰어를 1명 이상 지정한다.

리뷰어는 아래 내용을 확인한다.

- 변경 파일이 작업 범위와 맞는지
- 도메인형 구조를 지켰는지
- API 응답 형식을 지켰는지
- main 브랜치에 직접 반영되지 않았는지
- 테스트 또는 실행 검증이 되었는지
- 문서가 `docs/` 하위에 저장 또는 갱신되었는지

문제가 없으면 아래와 같이 리뷰한다.

```text
문제 없습니다.
잘 동작합니다.
```

문제가 있으면 에러 메시지, 로그, 캡처, 재현 방법을 남긴다.

## 14. 공통 API 응답 형식

모든 API 응답은 아래 형식을 따른다.

### 성공 응답

```json
{
  "isSuccess": true,
  "code": "common200",
  "message": "요청에 성공했습니다.",
  "result": {
  }
}
```

### 실패 응답

```json
{
  "isSuccess": false,
  "code": "TOKEN401",
  "message": "액세스 토큰이 만료되었습니다."
}
```

### Java 공통 응답 예시

```java
public record ApiResponse<T>(
    boolean isSuccess,
    String code,
    String message,
    T result
) {
    public static <T> ApiResponse<T> success(String code, String message, T result) {
        return new ApiResponse<>(true, code, message, result);
    }

    public static ApiResponse<Void> fail(String code, String message) {
        return new ApiResponse<>(false, code, message, null);
    }
}
```

## 17. 코드 컨벤션

### 이름 지정

| 대상 | 규칙 |
|---|---|
| 패키지명 | 소문자 |
| 클래스명 | PascalCase |
| 파일명 | PascalCase |
| DTO 클래스 | `UserDTO`처럼 DTO는 대문자 허용 |
| 메서드명 | camelCase |
| 변수명 | camelCase |
| 상수 | UPPER_SNAKE_CASE |
| enum 값 | UPPER_SNAKE_CASE |

예시:

```java
package com.moonju.preprocess.api.domain.user.entity;

public enum UserStatus {
    ACTIVE, DELETED, BANNED
}
```

### 포맷팅

- 들여쓰기: 4칸 공백
- 최대 줄 길이: 120자
- 파일 끝에 빈 줄 추가
- 중괄호는 같은 줄에서 시작
- 연산자 앞뒤 공백 추가

### 주석

- `//` 뒤에는 공백 하나 추가
- 5줄 이하는 `//` 사용 가능
- 긴 설명은 `/** */` 사용
- 불필요한 주석은 작성하지 않음

예시:

```java
// 사용자의 프로젝트 접근 권한을 검증한다.
public void validateProjectAccess(Long userId, Long projectId) {
    ...
}
```

## 19. Codex 작업 시작 프롬프트

Codex는 새 작업을 시작할 때 아래 원칙을 따른다.

```text
작업을 시작하기 전에 이슈를 먼저 생성해줘.
이슈 제목은 컨벤션에 맞춰 작성해줘.
작업 단위는 하나의 구현 단위로 쪼개줘.
main 브랜치에는 절대 push하지 말고, 이슈 번호 기반 브랜치를 만들어줘.
브랜치명은 타입/닉네임/이슈번호 형식으로 만들어줘.
작업 후 관련 명세는 docs/ 하위에 별도 md 파일로 저장하거나 기존 문서를 갱신해줘.
커밋 메시지는 [Feat], [Fix], [Docs] 같은 컨벤션을 지켜줘.
테스트나 빌드가 가능한 경우 반드시 실행하고 결과를 PR 본문에 적어줘.
PR은 컨벤션에 맞춰 생성하고 리뷰어가 확인해야 할 부분을 명시해줘.
PR을 만든 뒤에는 merge하지 말고 내가 검토할 수 있도록 링크만 알려줘.
```

## 26. 작업 완료 기준 Definition of Done

하나의 PR은 아래 조건을 만족해야 완료된 것으로 본다.

- [ ] 이슈가 존재한다.
- [ ] 브랜치명이 컨벤션을 따른다.
- [ ] main에 직접 push하지 않았다.
- [ ] 작업 범위가 하나의 구현 단위다.
- [ ] 커밋 메시지가 컨벤션을 따른다.
- [ ] 관련 명세가 `docs/` 하위에 저장 또는 갱신되었다.
- [ ] 테스트 또는 빌드 검증 결과가 PR 본문에 적혀 있다.
- [ ] 리뷰어가 확인해야 할 부분이 PR 본문에 적혀 있다.
- [ ] 사용자가 검토할 수 있도록 PR 링크를 제공했다.
- [ ] Codex가 임의로 merge하지 않았다.

## 29. 이번 프로젝트 특화 주의사항

- Worker는 image-test 레포의 전처리 메커니즘을 기준으로 한다.
- 전처리 단계는 최소한 아래 구조를 가진다.
    - Decode
    - Color Normalize
    - Orientation Normalize
    - Deskew
    - Crop
    - Denoise
    - Contrast Normalize
    - Binarization
    - Morphology Cleanup
    - DPI Normalize
    - Optional Sharpen
- API 서버는 전처리 작업을 직접 수행하지 않고 RabbitMQ 메시지만 발행한다.
- Object Storage 경로는 원본, 결과, preview, report, debug artifact를 분리한다.
- Google/Kakao 소셜 로그인 구조를 고려한다.
- Swagger 공통 응답 형식을 유지한다.
- `docs/` 문서는 구현과 함께 계속 갱신한다.
